# P4.5 Phase 方案：Radar Scored Analyst Writer

## 状态

ACTIVE

## 工程入口

- Python package: `backend/src/onlybtc/p45`
- Phase boundary: `onlybtc.p45.boundary.phase_boundary()`
- Legacy P4 package: `backend/src/onlybtc/p4`，仅保留历史回看、调试和手工对比用途。

## Phase 定位

P4.5 是 P3 与 P5 之间的新轻量推理/写作层，用来替代当前 P4 中过重的 Agent、交叉质询、Judge、Adversarial Review 和复杂发布门控叙事。

新的职责边界：

```text
P1: 真实数据采集、清洗、质量、source health
P2: Radar 模块聚合、模块趋势、模块质量
P3: 指标级正/零/负评分、历史窗口、事件窗口、反证与质量约束
P4.5: 4 个分析员中文评论 + 最终综合中文研究文章
P5: Dashboard 展示文章、评分、证据和附录
```

P4.5 不再承担“裁判系统”职责，而是做“有约束的研究写作器”：LLM 只解释已经量化好的证据，不自由裁判数据。

## 为什么要新建 P4.5

当前 P4 的问题不是功能不够，而是职责过多：

- Agent runtime、CrossExam、Revision、Judge、Adversarial Review、Final Controller 全部叠在一层。
- LLM 输入太像审计日志，导致输出也像审计日志。
- publish/block/runtime/fallback 叙事压过了趋势洞察。
- 同一轮运行成本高、失败点多、repair 逻辑重。
- 文章生成依赖多级中间 JSON，任何一层 fallback 都会影响最终可读性。

P4.5 的目标是把复杂性前移到确定性代码：

- P3 给每个指标打分：正分、零分、负分、不可用。
- P3/P2 给每个 Radar 模块输出总分、强度、置信度和趋势方向：bullish、bearish、neutral、mixed。
- P4.5 只拿“指标值 + 指标解释 + 评分 + 模块趋势 + 提示词框架”写人类可读评论。

## 新技术路径

### 1. P3 输出 Scored Metric Evidence

每个进入 P4.5 的指标必须包含：

```json
{
  "evidence_id": "ev-xxx",
  "radar_module": "event_policy",
  "metric_id": "macro_surprise_event_count",
  "source_id": "fxstreet-economic-calendar",
  "value": 0.0,
  "direction": "neutral",
  "metric_score": 0.0,
  "score_bucket": "zero",
  "quality_score": 0.7,
  "freshness_status": "fresh",
  "business_recency_status": "current",
  "metric_explanation": "事件政策 - 宏观惊喜事件计数；用于衡量本轮是否出现明显超预期宏观事件。当前为 0，说明没有形成直接宏观冲击。",
  "score_reason": "当前值为 0，未形成正向或负向趋势贡献。",
  "history_context": {
    "previous_value": null,
    "change_24h": null,
    "zscore": null,
    "percentile": null
  }
}
```

### 2. P4.5 构造 Analyst Evidence Pack

P4.5 按 4 个分析员切分 Radar 模块：

| 分析员 | Radar 模块 | 核心问题 |
| --- | --- | --- |
| macro_event_analyst | macro_radar, treasury_credit, asia_risk, event_policy | 宏观顺逆风、利率/美元/风险偏好、事件窗口 |
| liquidity_flow_analyst | dollar_liquidity, fund_flow, crypto_breadth | 美元流动性、ETF、稳定币、市场宽度 |
| microstructure_analyst | kline_orderflow, derivatives_crowding, trade_structure_flow, options_volatility | 短周期价格、成交量、OI、funding、期权、拥挤度 |
| onchain_structure_analyst | btc_total_state, btc_adoption, onchain_valuation | 链上估值、采用度、成本基础、BTC 内生结构 |

每个分析员输入只包含：

- 自己负责的 Radar 模块。
- 每个模块的总分、趋势方向、强度、置信度和质量。
- 每个模块的正分、负分、零分、不可用指标分布。
- 每个指标的值、评分、一句话解释。
- P3 事件窗口、反证、质量边界。
- 固定中文提示词框架。

Analyst Evidence Pack 应优先呈现模块级结论，再展开指标级证据：

```json
{
  "analyst_id": "macro_event_analyst",
  "modules": [
    {
      "radar_module": "macro_radar",
      "module_score": 0.18,
      "module_direction": "bullish",
      "module_strength": 0.42,
      "module_confidence": 0.74,
      "module_quality_score": 0.91,
      "score_bucket": "positive",
      "positive_metric_count": 5,
      "negative_metric_count": 2,
      "zero_metric_count": 6,
      "unavailable_metric_count": 1,
      "module_explanation": "宏观雷达总分为正，主要由 VIX 回落、Nasdaq 偏强和美元压力缓和贡献；但 OFR FSI 与部分利率指标仍形成抵消。",
      "metrics": []
    }
  ]
}
```

### 3. LLM 输出 Analyst Comment

每个分析员只输出一篇中文评论，不参与辩论：

```json
{
  "analyst_id": "macro_event_analyst",
  "title": "宏观与事件：风险偏好改善但政策窗口仍需观察",
  "direction_view": "mixed",
  "score_summary": "正分 3 项，负分 2 项，零分 7 项，模块综合偏混合。",
  "article": "中文评论正文，必须引用 evidence_id。",
  "key_positive_evidence_ids": [],
  "key_negative_evidence_ids": [],
  "neutral_watch_evidence_ids": [],
  "watch_items": [],
  "data_boundary": []
}
```

### 4. Final Writer 汇总 4 篇评论

Final Writer 不再做裁判辩论，只做综合写作：

1. 核心结论。
2. 四大板块共振与分歧。
3. 正分指标说明。
4. 负分指标说明。
5. 零分/中性指标为什么仍需观察。
6. 24h / 3d / 7d 观察路径。
7. 数据质量与边界。

### 5. HTML 输出

P4.5 HTML 应以文章为主：

- 主文章在最上方。
- 4 个分析员文章分区展示。
- Radar/metric scored evidence 表放在附录。
- LLM 深度中文研报追加在尾部，不能覆盖 deterministic 基线文章、分析员文章、Evidence 附录或 JSON。
- 四分析师 LLM 板块深度分析追加在 LLM 主研报之后，不能覆盖已有内容。
- JSON 只作为折叠审计，不进入正文主视觉。

## 清理旧 P4 的原则

因为项目已备份，P4.5 可以清理当前 P4 不需要的链路，但应分阶段：

1. 先新增 P4.5 并跑通，不立即删除 P4。
2. P4.5 验证稳定后，把旧 P4 标记 legacy。
3. 删除或隔离旧 P4 的 Agent/CrossExam/Judge/Adversarial Review runtime。
4. 保留可复用的：
   - evidence extraction 思路
   - HTML report 基础工具
   - LLM provider client
   - SQLite snapshot/history 写入能力
5. P5/P6 后续只消费 P4.5 文章和 scored evidence，不再依赖 P4 debate 表。

## 与 P3 的契约

P4.5 强依赖 P3-C16：

- P3 必须输出指标级 `metric_score`。
- P3 必须输出 `score_bucket = positive | zero | negative | unavailable`。
- P3 必须输出 Radar 板块级 `module_score / module_direction / module_confidence / module_explanation`。
- P3 必须输出 `metric_explanation` 和 `score_reason`。
- P3 必须保留 `evidence_id` 与 `metric_id/source_id/value` 的可追溯关系。
- P3 必须区分：
  - 当前 run 生产数据
  - 历史上下文
  - fallback/provider_required/unavailable

P4.5 不自己发明分数，不自己重算指标，只解释 P3 的评分。

3. P4.5-C02：Scored Evidence Pack Builder。
4. P4.5-C03：指标一句话解释词典与动态说明生成。
5. P4.5-C04：4 个 Analyst Writer 提示词与输出 schema。
6. P4.5-C05：Final Writer 综合文章生成。
7. P4.5-C06：P4.5 HTML 研究报告与证据附录。
8. P4.5-C07：旧 P4 legacy 隔离与 CLI/API 切换。
9. P4.5-C08：P1-P2-P3-P4.5 全链条真实重跑与验收。
10. P4.5-C09：LLM Research Writer 接入与全量 Evidence 深度中文研报生成。

13. P4.5-C12：HTML Report v2 决策视图、周期视图、反证条件与发文层渲染。

## 需要特别注意的细节

- 文章要允许“仓位、杠杆、拥挤度、资金流”等市场结构语言，但禁止交易指令。
- `zero` 分不是“无意义”，而是“当前不贡献方向，但可能是观察项”。
- 数据质量问题只放在数据边界，不要淹没主文章。
- LLM 失败时可以用 deterministic fallback 生成短文，但必须在 HTML 明确标注。
- LLM Research Writer 失败时不能 silent fallback；应写入 `status=failed` 和错误原因，并保留旧报告。
- Research Report V2 应保留 v1 字段，新增结构化决策卡、周期视图、反证条件和发文层。
- P1/P2/P3 暂缺 freshness、horizon、duplicate 字段时，P4.5 v2 先输出 warning，不阻塞报告生成。
- 同一个 evidence_id 必须能从正文跳到附录。
- 每个分析员都必须引用本模块 evidence，不允许跨模块乱引用。
- Final Writer 必须引用 4 个分析员结论和关键 evidence。
