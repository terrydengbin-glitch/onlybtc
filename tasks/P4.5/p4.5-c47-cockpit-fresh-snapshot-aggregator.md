# P4.5-C47 BTC Cockpit Fresh Snapshot Aggregator

Status: DONE

## Background

Once Radar Runtime refreshes modules at different cadences, the BTC center card can no longer assume that all 14 modules share the same timestamp. The cockpit must consume the latest valid module snapshots and apply freshness gates.

## Goal

Upgrade BTC Trend Cockpit aggregation:

```text
latest radar module snapshots
  -> freshness gate
  -> cadence role and horizon weighting
  -> BTC cockpit runtime snapshot
  -> dashboard / overview / history pass-through
```

## Aggregation Rules

```text
fast modules:
  sensitivity, early_warning, fast_signal

confirmation modules:
  confirmed_signal permission

regime modules:
  background bias, confidence cap, why_not_stronger

event_window:
  emergency overlay only, no direct radar score impact
```

## Suggested Output

```json
{
  "radar_runtime_context": {
    "runtime_snapshot_id": "",
    "asof_ts": "",
    "fresh_modules": 0,
    "stale_modules": 0,
    "blocked_modules": 0
  },
  "module_contribution_matrix": [],
  "freshness_summary": {},
  "why_not_confirmed": [],
  "runtime_data_quality_flags": []
}
```

## DoD

1. BTC cockpit no longer requires all modules to share one timestamp.
2. Every consumed module has snapshot_id, asof_ts, freshness_state, and age_sec.
3. Stale modules cannot participate in confirmed_signal.
4. If confirmation modules are stale, the center card max stage is watch or fast_signal.
5. Stale regime modules lower confidence only.
6. Output includes freshness_summary, module_contribution_matrix, and why_not_confirmed.
7. dashboard/latest, overview/latest, and history can pass through runtime cockpit.
