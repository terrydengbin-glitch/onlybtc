# P4.5-C42 Invalidation Workbench v2 规则生成器

状态：DONE

## 背景

当前 `Invalidation / Confirmation` 页面主要消费 `invalidation_rules` 与 `confirmation_rules`，本质仍是 P4.5 旧规则列表。随着 14 个 radar modules 已经升级为 BTC response、residual、signal_stage、support/pressure/conflict drivers 和 data quality 分层输出，Workbench 需要升级为 BTC Trend Cockpit 后面的动态证伪层。

## 目标

新增 `p45.invalidation_workbench.v2` generator，使其从 `btc_trend_cockpit.v2` 与 14 个 radar module signals 中生成动态验证结果：

```text
14 radar modules
  -> module evidence matrix
  -> BTC response / residual / micro response gate
  -> confirmation / invalidation rule state machine
  -> validation_state: confirmed | watching | refuted | conflict | blocked
```

Workbench 不负责重新预测 BTC 方向，而是验证当前 BTC thesis 是否被确认、证伪、冲突、阻断或仍在等待。

## 输入

```text
final_payload.btc_trend_cockpit
final_payload.btc_trend_cockpit.module_signals
final_payload.invalidation_rules
final_payload.confirmation_rules
final_payload.contract_validation
final_payload.data_quality
radar module payloads

```

## 硬规则

1. 模块压力不等于趋势确认。
2. 没有 BTC response，不允许 `triggered` confirmation / invalidation。
3. 没有 residual，不允许 `triggered` confirmation / invalidation。
4. 单一 radar module 不能触发 confirmed / refuted。
5. `funding`、`OI`、`ETF flow`、`macro`、`liquidation` 不允许单独触发裁决。
6. `pressure_score` 高但 BTC 不跌，只能 `armed` 或 `watching`。
7. `support_score` 高但 BTC 不涨，只能 `armed` 或 `watching`。
8. `data_quality_penalty >= 50` 或 contract blocking 时必须 `blocked`。

## DoD

1. run once 后 `final_payload.invalidation_workbench.schema_version = p45.invalidation_workbench.v2`。
2. 输出 `validation_state`、`validation_reason`、`scores`、`btc_response`、`module_evidence_matrix`、`rule_groups`、`timeline`。
3. neutral / watch_only 场景输出 `break_neutral_scenarios`，不生成“反证多头/反证空头”的错误语义。
4. bearish_watch / bullish_watch 场景分别生成确认与反证规则。
5. 缺 BTC response 或 residual 时，不出现 `triggered_rules`。
6. 单一 module 无法触发 confirmed / refuted 的单元测试通过。
7. data quality blocking -> `validation_state = blocked` 的测试通过。
8. P4.5 targeted pytest 通过。
