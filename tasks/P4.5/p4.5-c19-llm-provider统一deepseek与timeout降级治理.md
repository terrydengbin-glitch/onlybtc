# P4.5-C19 LLM Provider 统一 DeepSeek 与 Timeout 降级治理

## 状态

DONE

## 所属 Phase

P4.5 Radar Scored Analyst Writer

## 任务目标

统一 P4.5 LLM 输出链路默认使用 DeepSeek，降低多 provider API key 管理成本，同时补强 timeout、耗时记录、失败降级和 HTML 审计展示，确保 DeepSeek 超时或失败时不会阻断 deterministic P1/P2/P3/P4.5 主报告。

本任务主线范围是 P4.5：

- LLM Research Writer
- 四分析师 LLM Writer
- `p45-full-audit-with-llm` 一键全链条命令
- P4.5 HTML LLM 附录展示

旧 P4 Agent / CrossExam / Judge / Adversarial Review 已标记为 Legacy。若需要统一旧 P4 provider，作为兼容清理项处理，不影响 P4.5 主线验收。

## 背景

当前 P4.5 主线已经基本使用：

```env
ONLYBTC_P45_RESEARCH_PROVIDER=deepseek
ONLYBTC_DEEPSEEK_MODEL=deepseek-reasoner
```

P4.5 LLM 调用主要包括：

1. LLM Research Writer 主研报
2. 四分析师 LLM 板块分析

一键链路 `p45-full-audit-with-llm` 每次完整运行通常会触发 5 次 DeepSeek 调用：

- 1 次主研报
- 4 次分析师文章

近期真实 live 验收显示这 5 次调用可以完成，没有出现 timeout。但随着 Evidence 数量增加、prompt 变长、网络波动或 DeepSeek 高峰期排队，仍需要明确治理：

- timeout 不应太短。
- LLM 失败不应阻断 deterministic 报告。
- HTML 需要清楚显示哪些 LLM 成功、失败、耗时、provider、model。
- 一键命令 summary 需要明确 `completed` 或 `completed_with_llm_errors`。

## 风险评估

统一 DeepSeek 的好处：

- 少维护多套 API key。
- 输出风格更一致。
- 排障更简单。
- 一键链路更容易记录耗时和失败。
- 中文金融研报输出质量足够当前 P4.5 使用。

潜在风险：

- `deepseek-reasoner` 推理时间可能较长。
- 高峰期可能出现 timeout 或 rate limit。
- 所有 LLM 风格会趋同，少了多模型交叉视角。
- DeepSeek 不可用时，整个 LLM 附录层受影响。

治理原则：

- deterministic P1/P2/P3/P4.5 是主链，必须优先成功。
- LLM 是增强层和解释层，失败只能降级，不应阻断主报告。
- P4.5 HTML 必须明确区分 final_view 与 LLM internal_reference。

## 实施范围

1. 配置默认值统一
   - 确认 P4.5 默认：

```env
ONLYBTC_P45_RESEARCH_PROVIDER=deepseek
ONLYBTC_DEEPSEEK_MODEL=deepseek-reasoner
```

   - 建议调整：

```env
ONLYBTC_P45_RESEARCH_TIMEOUT_SECONDS=180
ONLYBTC_P45_RESEARCH_MAX_RETRIES=1
```

   - `.env` 示例和开发文档同步说明 DeepSeek 是 P4.5 默认 provider。

2. 一键链路耗时与状态记录
   - `p45-full-audit-with-llm` 输出中增加：
     - `llm_research_latency_ms`
     - `llm_analyst_total_latency_ms`
     - `llm_total_latency_ms`
     - `llm_provider`
     - `llm_model`
   - 对四分析师分别记录 status/error/latency。

3. Timeout / 失败降级
   - 如果 LLM Research Writer 失败：
     - 返回 `completed_with_llm_errors`
     - deterministic 报告继续可用
     - HTML LLM 主研报区显示失败原因
   - 如果四分析师 LLM 部分失败：
     - 已成功的分析师文章保留
     - 失败分析师显示 error 卡片
     - summary 记录 `analyst_failed_count`
   - 如果 DeepSeek key 缺失：
     - 支持 `--skip-llm`
     - 或返回明确 `Provider deepseek has no API key configured`

4. HTML 审计展示
   - LLM 附录区显示：
     - provider
     - model
     - runtime_mode
     - status
     - latency_ms
     - error
   - 一键链路 summary 可在 HTML 或 JSON 中追溯。

5. Legacy P4 兼容清理
   - 检查 `.env` 中旧 P4 provider：

```env
ONLYBTC_P4_LEVERAGE_MICROSTRUCTURE_PROVIDER=volcano
```

   - 如果用户确认统一 legacy provider，可改为：

```env
ONLYBTC_P4_LEVERAGE_MICROSTRUCTURE_PROVIDER=deepseek
```

   - 但旧 P4 已是 Legacy，不作为本任务 DoD 的阻断项。

## DoD

- P4.5 默认 LLM provider 明确为 DeepSeek。
- timeout 默认或建议值提升到适合 `deepseek-reasoner` 的区间。
- `p45-full-audit-with-llm` summary 包含 LLM provider/model/latency/error 汇总。
- DeepSeek 失败时返回 `completed_with_llm_errors`，deterministic 报告仍生成。
- HTML LLM 附录能展示 provider/model/status/latency/error。
- `--skip-llm` 仍可用。
- 单元测试覆盖：
  - 正常 LLM completed
  - LLM research failed
  - 四分析师部分 failed
  - skip_llm
- `pytest` 通过。
- `ruff check src tests` 通过。
- 真实 live 一键链路至少跑一次：

```powershell
python -m onlybtc.cli p45-full-audit-with-llm --run-mode live --runtime-mode deterministic --llm-runtime-mode llm
```

## 备注

本任务不是要恢复旧 P4 多 Agent 复杂链路，而是让 P4.5 的 LLM 增强层更简单、更稳定、更容易审计。DeepSeek 是默认主 provider；未来如果需要多模型复核，应单独作为“多模型对照验证”任务，而不是混入主生产链路。
