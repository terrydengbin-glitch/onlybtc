# P4.5-C18 Report v2 最后契约打磨与 LLM 附录治理

## 状态

DONE

## 所属 Phase

P4.5 Radar Scored Analyst Writer

## 任务目标

在 P4.5 Report v2 已经完成 final_view 一致性、决策卡、聚合审计、时间尺度、反证规则、发文层和一键全链条运行之后，进行最后一轮小修，提升审计体验与长期维护稳定性。

本任务不改变 P1/P2/P3 数据链路，不改变 P3 指标评分和 P4.5 final_view 聚合算法，只优化 P4.5 报告契约与 HTML 展示层。

## 背景

当前 P4.5 Report v2 已经基本达到稳定版：

- `final_view` 已成为唯一主裁判结论。
- `legacy_core_view` 已降级为历史兼容字段。
- `decision_card`、`aggregation_audit`、`research_article`、`publish_article` 已统一口径。
- `why_not_strong` 非空。
- `invalidation_rules` 已结构化并可读渲染。
- `score_normalization` 已解释 raw score 到 directional score 的归一化。
- 一键命令 `p45-full-audit-with-llm` 已能跑完整 P1/P2/P3/P4.5 + LLM 主研报 + 四分析师 LLM。

剩余问题属于边角契约和审计体验：

1. 同一 horizon 的 `support_drivers` 或 `pressure_drivers` 内部可能出现重复 `metric_id`。
2. 同一指标跨模块出现时，HTML 展示未按 `duplicate_group_id` 做聚合或去重说明。
3. Final JSON 已有 `confirmation_rules`，但 HTML 主要展示反证条件，缺少“偏空确认条件”区块。
4. LLM 深度研报仍可能包含 raw evidence_id，应明确标记为内部解释参考，默认折叠，避免被误读为最终结论或外发内容。
5. 四位 deterministic 分析员文章更像审计附录，应在标题和位置上明确“审计附录”属性。

## 实施范围

1. Horizon driver 去重
   - `support_drivers` 内部按 `metric_id` 去重。
   - `pressure_drivers` 内部按 `metric_id` 去重。
   - 同一 `duplicate_group_id` 多次出现时，保留绝对贡献最大的代表项，并在字段中保留合并说明。
   - 不改变底层 Evidence 附录，Evidence 附录仍保留所有原始 evidence。

2. 新增 contract checks
   - `horizon_driver_unique_within_side`
   - `horizon_driver_unique_by_duplicate_group`
   - `has_confirmation_rules_rendered`
   - `llm_article_scope_declared`

3. Confirmation Rules HTML 渲染
   - 在反证条件之后新增区块：
     - `偏空确认条件 / Confirmation Rules`
   - 使用与反证条件相同的表格渲染方式。
   - `conditions` 渲染为：

```text
etf_net_flow.metric_score < 0 AND taker_buy_sell_ratio.metric_score < 0
```

   - `action_if_triggered` 渲染为：

```text
neutral -> weak_bearish
```

4. LLM 深度研报附录化
   - 在 LLM Research payload 或 HTML 展示层声明：
     - `llm_article_scope = internal_reference`
     - 不参与 `final_view`
     - 不作为发文版本
   - HTML 中 LLM 深度研报默认折叠。
   - 如果 LLM 观点与 final_view 冲突，继续保留冲突提示。

5. 四位分析员文章附录化
   - 将 deterministic 四位分析员文章区块标题调整为：
     - `四位分析员审计附录`
   - 默认折叠或明显放在审计区域。
   - 保留 evidence_id 引用能力，但避免误读为正式发文。
