# P5-C03 Radar 拓扑节点动态连线、拖拽布局与数据源状态展示

## 状态

DONE

## 当前架构对齐（2026-05-22）

Radar 节点必须对齐当前 14 个 P2 Radar module 和 P3/P4.5 scored evidence。节点数据来自 `GET /api/p45/radar-modules/latest` 与 `GET /api/p45/radar-modules/{module_id}`。

每个节点必须包含：`radar_module`、`module_score`、`module_direction`、`module_strength`、`module_confidence`、`module_quality_score`、positive/negative/zero/unavailable count、`module_explanation`、`data_boundary`、top support/pressure drivers。

每个 metric 必须可下钻到：`metric_score`、`metric_effective_score`、`freshness_weight`、`horizon_weight`、`duplicate_adjustment`、`horizon_tags`、`duplicate_group_id`、`source_ts`、`collected_at`、`freshness_minutes`、`is_stale`、`p45_metric_brief`。

### 当前发现的问题（2026-05-22）

P5-C01/P5-C02 后，BTC 中心节点已变为真实状态卡，但 Radar box 仍由 CSS 绝对定位，连线仍使用固定 SVG path。节点尺寸、文本、中心节点高度变化后，出现 box 与连线脱节、连线穿过卡片、局部节点遮挡的问题。

本卡需要把 Radar 拓扑升级为可维护的动态图：

- box 位置使用数据化 layout，而不是散落 CSS 坐标。
- line/path 根据 box 与 BTC 节点实际位置动态生成。
- 桌面支持鼠标拖拽 Radar box，拖拽时连线实时跟随。
- 拖拽位置保存到 `localStorage`，刷新后保留；提供 `Reset Layout` 恢复默认布局。
- 移动端保持线性卡片布局，不强制拖拽。
- 拖拽仅影响 UI layout，不得影响 P2/P3/P4.5 权重、方向、评分或结论。

### 动态焦点缩放与 BTC protected zone（2026-05-22）

本卡同时引入“中心不可覆盖 + 视觉远近感”的交互规则：

- BTC 中心节点固定，不可拖拽、不缩放。
- Radar 节点拖拽时不得覆盖 BTC 中心卡片的 protected zone。
- protected zone = BTC node DOM rect + 12px padding；Radar box 最多吸附到 protected zone 外沿。
- Radar 节点根据离 BTC 中心的距离动态缩放，越靠近 BTC 视觉越大，越远视觉越小。
- 建议 scale 范围：`0.78 <= scale <= 1.08`；第一版可使用线性距离映射。
- scale 只表达 UI focus/depth，不代表业务权重、交易强度或评分重要性。
- 连线 `opacity` / `strokeWidth` 可随距离调整：越近越亮、越粗；越远越淡、越细。
- 第一版只实现 BTC protected zone，不做 Radar 节点之间复杂碰撞；若节点之间仍重叠，后续再做 repel / collision。
- `localStorage` 只保存 `{ x, y }`，不保存 scale；scale 每次由当前位置实时计算。

### C03 补充修正：拖动联动与 Radar 可读摘要（2026-05-22）

当前第一版实现了拖动目标 box、动态连线和距离缩放，但还需要补齐两项体验：

1. 拖动一个 Radar box 靠近 BTC 时，其他 Radar box 也需要产生轻量动态让位，而不是只改变缩放。
2. Radar box 小字不应显示数据 dump，例如 `+4 / -0 / 0=4 / NA=2 · q=0.8226` 作为主文案；主文案应是一句人能读懂的模块摘要。

处理方案：

- 拖动节点为 active node；其他节点根据 active node 与 BTC 的距离产生轻量 repel。
- repel 方向以“远离 active node，同时不进入 BTC protected zone”为主。
- repel 位移只作用于 UI layout，默认不超过初始位置的 `6%-10%`，避免布局失控。
- 松手后保存当前布局；`Reset Layout` 恢复所有节点默认位置。
- Radar box 主文案优先级：`module_explanation` -> `module_summary` -> 前端根据 `module_id + direction + counts + quality` 自动生成中文摘要。
- 审计数据只作为 secondary meta 显示，例如 `q=0.82 · +4/-0 · NA=2`，不作为主解释。
- 完整 metrics、freshness、duplicate、horizon 仍放在 Radar Detail / Evidence 页面展示。
- 本修正不回 P2/P3/P4.5 修改字段；后续如需固化正式字段，可另开 P4.5 任务生成 `module_summary_cn`。

## UI / 高保真对齐（2026-05-22）

Radar 节点必须对齐 `ui-references/p5-dashboard-high-fidelity.html` 中 `.node` 和 `.links` 的视觉语义：

- 节点方向类：`support`、`pressure`、`mixed`、`quality`。
- 每个节点显示 module 名称、方向 pill、一句话 module explanation、score bar、score 数字。
- 连线颜色必须跟 module dominant side 一致：support=青绿，pressure=红，mixed=黄虚线，quality/fallback=紫。
- 节点数量必须来自 API 返回，默认布局可使用固定 14 点位映射，但映射必须按 `module_id` 可维护配置，不允许硬编码业务值。
- 桌面端节点可拖拽，拖动时连线跟随节点边缘或中心锚点更新，不允许继续使用写死 SVG path。
- 节点拖拽需要限制在 `.topology` 容器内，避免被拖出画布。
- 节点拖拽需要限制不能覆盖 BTC protected zone，靠近中心时自动吸附在 BTC 卡片外沿。
- Radar 节点靠近 BTC 时动态放大，远离 BTC 时动态缩小，形成轻量 3D depth 效果。
- 拖动一个 Radar 节点时，其他 Radar 节点应按轻量 repel 规则产生联动位移，形成空间让位感。
- 提供 `Reset Layout` 控件，清理本地布局并恢复默认 14 点位。
- `Reset Layout` 必须作为明确的复原键展示在拓扑图工具区，用户点击后立即回到默认位置。
- 复原键只清理 `onlybtc:p5:radar-layout:v1` 本地布局，不触碰 API 数据、run history、评分、权重或 P4.5 结论。
- 点击节点打开 Radar Detail，点击 score/evidence chip 可进入 Evidence 筛选。

## 实施范围

- 14 个 Radar module 全量展示，不允许遗漏新增进入 Radar 的采集指标。
- 每个 Radar 节点显示 signal、strength、confidence、data_quality、quality_score、feature_count。
- 显示 `feature_run_scope`、`historical_fallback`、`source_resolution`、`fallback_used`、`conflict_count`。
- 支持点击进入 Radar Detail，并带上 `module_id`、`p2_radar_run_id`、`collect_run_id`。
- 使用 `GET /api/p45/radar-modules/latest` 渲染节点列表，不使用静态假数据。
- 使用 `GET /api/p45/radar-modules/{module_id}` 打开模块详情，展示 module summary、metric evidence、freshness、duplicate group、horizon 信息。
- 使用 Evidence API 或前端已加载 evidence payload 支持 score/evidence chip 下钻。
- 实现可拖拽 layout：`pointerdown` / `pointermove` / `pointerup`，拖拽时禁用文本选择。
- 动态生成 SVG path：根据每个 Radar 节点与 BTC 中心节点当前 DOM rect 计算锚点，线条颜色和虚实由 API direction / dominant side 驱动。
- 实现 `protectedRect` 计算：读取 BTC center node 的 DOM rect，并将 Radar box 拖拽目标 clamp 到 protected zone 外部。
- 实现 `scaleByDistance` 计算：根据 Radar 节点中心点到 BTC 节点中心点距离，生成 CSS transform scale。
- 实现 `lineDepthByDistance` 计算：根据距离生成 SVG path 的 opacity / strokeWidth。
- 实现 `repelByActiveNode` 计算：拖动 active Radar 节点时，其他节点根据距离产生轻量位移，位移幅度受上限约束。
- 实现 `moduleReadableSummary`：把 `module_explanation`、`module_summary` 或前端 fallback 中文摘要作为 Radar box 主文案。
- 实现 `moduleAuditMeta`：把质量分、正负零/不可用计数作为 secondary meta，而不是主文案。
- `localStorage` key 建议：`onlybtc:p5:radar-layout:v1`，保存 `{ module_id: { x, y } }`。
- 提供 `Reset Layout` 复原键，清理本地 layout 并重新应用默认布局；复原后所有 box scale 与 line depth 立即重新计算。
- 遵守 evidence + data、历史窗口、数据质量、反证机制和预警边界。
- 不输出交易建议，不引入开仓、止损、仓位或杠杆逻辑。

## 验收标准

- 与《开发文档.md》的总体架构一致。
- 14 个 Radar module 都有节点。
- Dashboard Radar 节点与高保真相对布局一致，桌面截图中所有节点、连线和文字不重叠。
- 连线不再使用固定 SVG path；节点位置变化后，连线必须实时跟随。
- 桌面端可拖拽 Radar box，拖拽过程连线实时更新。
- Radar box 不能拖出 `.topology` 容器。
- Radar box 不能覆盖 BTC 中心卡片；靠近 BTC 时必须停在 protected zone 外侧。
- Radar box 根据距离 BTC 的远近动态缩放，且 BTC 中心卡片保持大小不变。
- 拖动一个 Radar box 靠近 BTC 时，其他 Radar box 产生轻量远离/让位，视觉上形成远近变化。
- 其他 Radar box 的联动位移有上限，不得无限漂移或离开 topology 容器。
- 动态缩放不得影响 API score、module weight、排序或 P4.5 final_view。

- 刷新后保留用户拖拽布局。
- `Reset Layout` 可恢复默认布局。
- 复原键点击后，所有 Radar box 回到默认 14 点位，连线、缩放、透明度同步回到默认状态。
- 复原键不得触发全链条 run，不得修改 SQLite / HTML 报告 / P4.5 payload。
- 节点方向、score bar、连线颜色必须由 API 字段驱动，不允许静态样例数据残留。
- Radar 的数据质量降权、历史 fallback、多源冲突必须可见。
- 点击 Radar 节点必须调用或消费 `GET /api/p45/radar-modules/{module_id}`，并能看到该模块 metrics / evidence / freshness / duplicate / horizon 字段。
- 点击 Evidence/score 入口必须能进入 Evidence 页面或筛选上下文。
- Radar box 主文案必须是中文可读摘要，不允许把 `+/-/0/NA/q` 数据串作为主文案。
- Radar box 可以显示 secondary audit meta，但必须弱化，不影响主阅读。
- 关键状态、错误和数据质量可观测。
- 不绕过 P4.5 final_view、反证/确认规则、预警等级或数据质量约束。
