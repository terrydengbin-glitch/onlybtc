# P5-C31 前端多接口加载容错与空态治理

## Phase

P5 Dashboard 与可视化层

## 背景

Dashboard 偶发出现 `14 Radar modules · 0 scored evidence`、BTC 中心卡片停留在 `Waiting for P4.5 decision card` 的空态。排查发现 FastAPI 的核心接口本身有数据，但前端 `refreshLatest()` 使用 `Promise.all` 同时请求多个接口，只要任一非核心接口临时失败，整批状态都不会写入，导致页面退回默认空态。

## 目标

- 将 Dashboard latest 数据加载从“一处失败全量失败”改为“分接口容错加载”。
- 核心数据 Dashboard / Overview / Radar Modules / Evidence 成功时必须优先渲染。
- 非核心接口失败只记录错误，不清空已成功数据，不阻断拓扑图和 BTC 决策卡。
- 历史模式仍保持不刷新 latest。

## 范围

- `frontend/src/store.ts`
- P5 API client/store DoD

## DoD

- `refreshLatest()` 不再使用会整体短路的 `Promise.all` 主加载逻辑。
- 单个 endpoint 失败时，其他成功 endpoint 仍写入 state。
- `state.errors` 能记录失败 endpoint。
- 若已有上一轮成功数据，新一轮非核心失败不会把页面打回默认空态。
- Dashboard contract validator 通过。
- Vue build 通过。

## 测试

- `npm run build`
- `python scripts/validate_p5_dashboard_contract.py`
- `python scripts/validate_p5_page_dod.py`

## 完成记录

- 已将 `refreshLatest()` 从整体短路式 `Promise.all` 改为 `Promise.allSettled` 分接口加载。
- 已新增 `assignLatestPayload()`，让成功 endpoint 独立写入对应 state。
- 已新增 `rememberEndpointError()`，非核心接口失败时只进入 `state.errors`，不阻断核心 Dashboard 渲染。
- Dashboard / Overview / Radar Modules / Evidence 标记为核心接口；若首次加载核心数据也失败，才显示核心错误。
- 已保留上一轮成功 state，避免某个可选接口失败时页面退回 `0 scored evidence` / `Waiting for P4.5 decision card` 的假空态。
