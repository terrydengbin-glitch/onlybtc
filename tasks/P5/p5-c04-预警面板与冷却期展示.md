# P5-C04 预警面板与冷却期展示

## 状态

DONE

## 当前架构对齐（2026-05-22）

预警面板优先消费 P3 输出，包括 alerts、event windows、watch rules、post-event 风险总结。P4.5 的 `invalidation_rules` 和 `confirmation_rules` 作为观察条件展示，不等同于交易信号。

FastAPI 读取：`GET /api/p3/alerts/latest`、`GET /api/p45/invalidation/latest`。每条预警必须带同 run lineage：`collect_run_id / p2_radar_run_id / p3_run_id / final_run_id`。

## 所属 Phase

P5

## 任务目标

实现 Dashboard 预警面板，展示 P3 info/watch/warning/critical、去抖动、冷却期、升级/降级条件和 P4 发布约束。

## 背景依据

- [开发文档.md](../../开发文档.md)
- [task index.md](../../task%20index.md)

## 实施范围

- 展示活跃预警、冷却期状态、升级/降级原因、关联 evidence、关联 invalidation。
- 事件窗口和 post-event 风险总结必须能进入预警面板。
- 人工处置只影响运营状态，不改写原始审计记录。
- 遵守 evidence + data、历史窗口、数据质量、反证机制和预警边界。
- 不输出交易建议，不引入开仓、止损、仓位或杠杆逻辑。

## 输入

- P9-C05 `GET /api/alerts`
- P3-C06/P3-C07/P3-C14 输出
- P4 Final Controller `blocked_by` 与 publish constraints

## 输出

- 可运行或可复用的代码、配置、Schema、接口、组件或文档。
- 必要的测试、验证记录或运行说明。

## 验收标准

- 与《开发文档.md》的总体架构一致。
- 预警必须可追溯到触发数据、冷却期、反证和下一步 watch 条件。
- 关键状态、错误和数据质量可观测。
- 不绕过 P4.5 final_view、反证/确认规则、预警等级或数据质量约束。
