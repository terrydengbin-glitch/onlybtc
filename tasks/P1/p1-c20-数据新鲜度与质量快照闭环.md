# P1-C20 数据新鲜度与质量快照闭环

## 状态

DONE

## 所属 Phase

P1 数据源与历史数据底座 / P8 SQLite 数据质量表

## 任务定位

补齐当前数据采集链路里的“新鲜度 freshness”与“系统级数据质量快照”闭环。

当前系统已经有：

- `MetricSample.quality_score`
- `metric_values.quality_score`
- `source_health_events`
- `data_quality_snapshots` 表结构
- radar confidence 对 `quality_score` 的初步消费

但还缺：

- 按 source/metric 的更新时间计算 freshness。
- 按 source 自己的 `refresh_minutes` 判断 stale/expired。
- stale 数据对 metric quality 和 radar confidence 的显式折扣。
- 每次 collect / run_once 后生成 `data_quality_snapshots`。
- Data Quality / Source Detail API 可直接读取的质量摘要。

## 核心原则

数据质量不能只看“这次请求有没有成功”，还必须看：

```text
数据是否足够新？
是否符合该 source 的刷新周期？
是否是 fallback / proxy？
是否有字段缺失？
是否长期没有成功更新？
```

##  freshness 定义

### Source refresh policy

优先级：

```text
1. source.metadata.refresh_minutes
2. settings.default_refresh_seconds
3. 默认 10 分钟
```

### Freshness status

```yaml
freshness:
  fresh:
    condition: age_seconds <= expected_refresh_seconds * 1.5
    quality_discount: 0

  stale:
    condition: age_seconds > expected_refresh_seconds * 1.5
    quality_discount: -0.15

  expired:
    condition: age_seconds > expected_refresh_seconds * 6
    quality_discount: -0.35

  missing:
    condition: no successful metric/raw observation exists
    quality_discount: -1.0
```

说明：

- `fresh`：可以正常参与雷达判断。
- `stale`：可以展示，但要降低 confidence。
- `expired`：默认不应用于强判断，只能作为历史参考。
- `missing`：指标不可用。


不新增重复表，除非现有字段无法表达。

### source_health_events payload 语义

当前 `SourceHealthEvent` 没有 metadata 字段，因此 freshness 先写入：

- `status`
- `quality_score`
- `latency_ms`
- `message`

message 建议包含机器可读前缀：

```text
freshness=fresh age_seconds=522 expected_seconds=600
freshness=stale age_seconds=1450 expected_seconds=600
freshness=expired age_seconds=7200 expected_seconds=600
```

如果后续需要更清晰，可升级 P8 schema，为 `source_health_events` 添加 `metadata_json`。

## 采集链路改造点

### 1. SourceConfig refresh policy

为所有 source 支持：

```yaml
metadata:
  refresh_minutes: 10
  stale_after_minutes: optional
  expired_after_minutes: optional
```

已有 source 没写时使用系统默认。

### 2. persist_collection_result

当前逻辑：

```text
quality = min(metric.quality_score)
status = quality_status(quality, latency_ms)
```

需要改为：

```text
freshness = compute_freshness(source, latest_metric_ts or observed_at, now)
quality = apply_freshness_discount(raw_quality, freshness)
status = quality_status(quality, latency_ms, freshness)
```

并把 freshness 写入 source health message。

### 3. historical_window

返回值增加：

```yaml
age_seconds: number
expected_refresh_seconds: number
freshness_status: fresh | stale | expired | missing
freshness_discount: number
effective_quality_score: number
```

`effective_quality_score` 应用于雷达，不直接覆盖原始采样质量。

### 4. radar service

雷达计算时使用：

```text
effective_quality_score
```

如果 metric 为 `expired`：

```text
available = false
invalidation_signals.stale_metrics += metric_id
```

如果 metric 为 `stale`：

```text
available = true
quality 降权
confidence 降权
```

### 5. data_quality_snapshots

每次 collect 或 run_once 后生成快照：

```yaml
data_quality_snapshot:
  run_id: optional
  score: 0.0-1.0
  status: healthy | warning | critical
  payload:
    source_count: 54
    healthy_sources: 48
    stale_sources: 3
    expired_sources: 1
    missing_sources: 2
    avg_quality_score: 0.82
    freshness_score: 0.88
    fallback_count_24h: 2
    rate_limit_count_24h: 0
    stale_metrics:
      - metric_id: ...
        source_id: ...
        age_seconds: ...
    module_discounts:
      - module_id: btc_adoption
        reason: stale lightning data
        discount: 0.12
```
