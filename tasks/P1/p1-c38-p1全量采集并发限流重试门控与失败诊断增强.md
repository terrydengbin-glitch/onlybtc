# P1-C38 / P1 全量采集并发限流、重试门控与失败诊断增强

## 状态

DONE

## 所属 Phase

P1 数据源接入与采集层

## 背景

真实全链条运行中出现 P1 全量采集不稳定：同一轮会同时拉起大量 HTTP/API/Playwright source，导致页面源 timeout、API 源短暂失败、错误消息为空，并且后续 P2/P3/P4.5 仍可能消费低质量 collect run。

## 目标

把 `collect_sources()` 从无控制的全并发 `asyncio.gather` 升级为可限流、可重试、可诊断、可门控的采集编排。

## 实施范围

1. 采集并发限流：
   - Playwright 页面源低并发。
   - HTTP/API/交易所源中高并发。
   - 官方日历/RSS/低频源中等并发。
2. Critical source 重试：
   - Binance 核心价格/Kline/Funding/OI/Taker。
   - FRED 核心宏观源。
   - Coingecko global。
   - official macro event calendar。
3. 错误诊断增强：
   - 空异常消息必须补异常类名。
   - 区分 timeout、network、rate_limited、access_denied、server_error。
   - raw payload 保留 collect_attempts 与 attempt_count。
4. P1 collect gate：
   - 输出 `collect_gate_status`、`collect_gate_reasons`、source 成功/失败/告警数量、critical failure 数量、本轮 metric 数量。
   - gate payload 写入 DataQualitySnapshot，供 P1-C22/P2/P3/P4.5/API 消费。

## DoD

- 全量采集不再一次性无控制并发所有 source。
- Critical source 短暂失败会自动 retry。
- 空 `error_message` 不再出现，至少包含异常类名和分类。
- collect result 与 data_quality snapshot 都包含 `collect_gate`。
- 单源调试路径继续可用。
- 单元测试覆盖 retry、空异常诊断、critical gate failed。
