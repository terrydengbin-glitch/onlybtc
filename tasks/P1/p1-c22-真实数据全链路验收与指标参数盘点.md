# P1-C22 真实数据全链路验收与指标参数盘点

## 状态

DONE

## 所属 Phase

P1 数据源与历史数据底座 / P2 全量雷达 / P8 SQLite 持久化层 / P5 Dashboard 数据质量页与证据页

## 任务定位

对当前系统做一次真实环境的端到端验收：真实拉取所有已接入数据源，写入 SQLite，执行多源仲裁，交给 P2 Radar 全量消费，并输出可审计的数据参数清单、失败清单和问题清单。

这不是新增单个数据源任务，而是一次“数据链条总验收”。目标是确认：

- 数据是否能真实获取。
- 数据是否能标准化为 `metric_values`。
- 每条数据是否具备 `run_id`、`source_id`、`metric_id`、`ts`、quality、freshness 等关键字段。
- SQLite 是否正确存储原始观测、标准指标、source health、data quality snapshot。
- 多源指标是否经过仲裁，而不是混源计算趋势。
- Radar 是否能消费正确指标，并生成 `radar_outputs`、`feature_values`、`module_json_outputs`。
- 不能采集的数据要列出原因，并区分是缺 key、需登录、需付费、页面反爬、源站暂不可用、解析失败，还是指标本身只能代理。

## 执行范围

### 1. 数据采集全量真实执行

执行全部 `SOURCE_CONFIGS` 中可真实运行的 source：

```powershell
..\.venv\Scripts\python.exe -m onlybtc.cli collect-sources --mode live
```

如果某些源需要分批执行，按数据源组拆分：

```powershell
..\.venv\Scripts\python.exe -m onlybtc.cli collect-sources --mode live --source-id <source_id>
```

必须记录每批：

- `run_id`
- source 数量
- 成功数量
- 失败数量
- 写入 `raw_observations` 数量
- 写入 `metric_values` 数量
- `data_quality_snapshot` 分数
- stale / expired / missing 源数量

### 2. SQLite 存储验收

验收表：

- `source_runs`
- `raw_observations`
- `metric_values`
- `source_health_events`
- `data_quality_snapshots`
- `radar_outputs`
- `feature_values`
- `module_json_outputs`

检查项：

- 同一轮采集的 `run_id` 是否贯穿采集相关表。
- 每条 `metric_values` 是否至少具备：
  - `metric_id`
  - `source_id`
  - `run_id`
  - `ts`
  - `value`
  - `quality_score`
- 每个 source 是否有 source health。
- 每轮采集是否有 data quality snapshot。
- 时间戳是否统一 UTC 存储，UI 层再转换为 UTC+8。

### 3. 多源仲裁验收

对存在多源的指标逐个执行：

```powershell
..\.venv\Scripts\python.exe -m onlybtc.cli metric-window <metric_id>
```

必须确认：

- 默认返回 selected source。
- `candidates` 列出所有候选源。
- `selected_reason` 可解释。
- 多源差异超过阈值时出现 `conflict.detected=true`。
- 指定 source 查询仍可工作：

```powershell
..\.venv\Scripts\python.exe -m onlybtc.cli metric-window <metric_id> --source-id <source_id>
```

重点指标：

- `btc_price`
- `lightning_capacity_btc`
- `stablecoin_supply`
- `exchange_balance_delta_1d_proxy`
- `realized_price`
- `sth_cost_basis`
- `lth_cost_basis`
- `macro_surprise_score`
- `fed_speech_risk`

### 4. Radar 全量消费验收

执行：

```powershell
..\.venv\Scripts\python.exe -m onlybtc.cli analyze-radars
```

检查每个 radar module：

- 是否有输出。
- `features` 是否覆盖该模块要求的指标。
- 缺失指标是否写入 `invalidation_signals.missing_metrics`。
- stale / expired 指标是否写入 `invalidation_signals.stale_metrics` / `expired_metrics`。
- 多源冲突是否写入 `conflicting_evidence.source_conflicts`。
- `data_quality` 是否合理反映指标覆盖率与质量。

必须覆盖雷达板块：

- BTC 总状态
- 宏观雷达
- 美元流动性雷达
- 美债/信用压力雷达
- 资金流雷达
- BTC 采用率雷达
- 链上估值与筹码雷达
- K 线盘口与价格结构雷达
- 衍生品拥挤度雷达
- 交易结构/链上/衍生品流量雷达
- 期权波动率雷达
- 加密市场广度雷达
- 亚洲风险雷达
- 事件/政策/Fed/宏观冲击雷达

### 2. 指标参数清单

清单表必须覆盖所有已注册指标。字段如下：

| 字段 | 说明 |
|---|---|
| `metric_id` | 系统内部参数名 |
| `display_name` | 中文/英文展示名 |
| `source_id` | 来源 ID |
| `source_name` | 来源名称 |
| `source_group` | 所属数据源组 |
| `radar_module` | 所属雷达板块 |
| `source_update_frequency` | 数据源自身更新频率，不是本地采集频率 |
| `local_refresh_policy` | 本地采集频率/刷新策略 |
| `source_form` | `raw_source` / `self_calculated` / `proxy` / `event_countdown` / `llm_scored` |
| `value_unit` | 单位 |
| `higher_is` | 上升含义：bullish / bearish / mixed / neutral |
| `quality_score` | 当前质量分 |
| `freshness_status` | fresh / stale / expired / missing |
| `age_seconds` | 当前数据年龄 |
| `expected_refresh_seconds` | 期望更新间隔 |
| `importance` | high / medium / low |
| `arbitration_role` | selected / candidate / single_source |
| `conflict_status` | none / low / medium / high |
| `sqlite_status` | stored / missing / partial |
| `radar_consumed` | yes / no |
| `notes` | 备注 |

### 3. 失败数据源与问题清单

失败清单必须按以下原因分类：

- `missing_api_key`: 缺 API key。
- `login_required`: 需要登录态。
- `paid_provider_required`: 需要付费授权。
- `anti_bot_or_captcha`: 反爬或验证码。
- `source_unavailable`: 源站不可用。
- `parse_failed`: 页面或 payload 解析失败。
- `rate_limited`: 限流。
- `proxy_only`: 只能用代理指标，无法代表精确指标。
- `not_implemented`: 已规划但未实现。

每条失败项必须包含：

| 字段 | 说明 |
|---|---|
| `metric_id` | 影响指标 |
| `source_id` | 失败数据源 |
| `radar_module` | 影响雷达 |
| `failure_reason` | 失败分类 |
| `detail` | 具体错误 |
| `impact` | high / medium / low |
| `fallback_or_proxy` | 当前替代方案 |
| `next_action` | 下一步建议 |

## 指标重要程度规则

### high

直接影响 BTC 中央判断或关键风险识别：

- BTC 价格与 K 线结构
- DXY / 美债收益率 / real yield / VIX
- Fed balance sheet / reserves / TGA / ON RRP
- ETF flow
- funding / OI / liquidation
- exchange flow / stablecoin liquidity
- MVRV / realized price / STH-LTH cost basis
- macro surprise / Fed speech risk / FOMC/CPI/PCE/NFP countdown

### medium

影响模块判断，但通常作为辅助确认：

- BTC dominance
- TOTAL2
- ETH/BTC
- Asia risk
- options IV / skew / put-call / max pain / gamma proxy
- miner flow proxy
- whale flow proxy
- lightning capacity

### low

更多用于背景、解释或 UI 展示：

- source metadata
- calendar low-impact events
- provider page health
- non-core breadth indicators
