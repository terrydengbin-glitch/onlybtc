# P1-C24 多源质量优先仲裁与冲突证据输出

## 状态

DONE

## 来源

```text
source_conflict: 5 metric rows
```

用户要求：

> 多源的以质量最佳的源为主，其他的源为 fallback。

## 所属 Phase

P1 数据源与历史数据底座 / P2 Radar / P5 Evidence 与 Radar Detail / P8 SQLite

## 当前问题

P1-C21 已实现多源仲裁，但当前规则顺序是：

```text
freshness > source priority > effective_quality > ts
```

这会导致在两个源都 fresh 时，优先级数字更小的源可能胜过质量更高的源。

现在需要按新的业务要求调整：

```text
多源默认以质量最佳的源为主；
其他源作为 fallback / cross-check；
如果质量接近，再看 source priority；
如果主源 stale/expired，再自动切换到 fresh fallback。
```

## 目标规则

### 默认仲裁规则

候选源排序：

```text
1. freshness_status: fresh > stale > expired > missing
2. effective_quality_score: 高者优先
3. source priority: 数字越小优先
4. ts: 更新者优先
```

解释：

- fresh 数据永远优先于 stale/expired。
- 在同样 fresh 的情况下，以质量最佳为主。
- source priority 用于质量相近时打破平局。
- 其他源标记为 `fallback` 或 `cross_check`，不能混入趋势计算。

### 候选角色

`historical_window(metric_id)` 返回：

```yaml
candidates:
  - source_id: xxx
    role: selected
  - source_id: yyy
    role: fallback
  - source_id: zzz
    role: cross_check
```

建议：

- selected：当前消费源。
- fallback：可接管的备源。
- cross_check：仅用于冲突证据，不自动接管。

### 冲突输出

当多源差异超过阈值：

```yaml
conflict:
  detected: true
  items:
    - metric_id: lightning_capacity_btc
      primary_source: clarkmoody-dashboard
      conflicting_source: mempool-lightning-network-stats
      relative_diff: 0.026
      severity: low
```

阈值保持：

- low：`> 2%`
- medium：`> 5%`
- high：`> 10%`

## P2 Radar 要求

Radar 默认只消费 selected source。

Radar output 必须写入：

- selected source
- candidates
- conflict
- selected_reason
- fallback 可接管状态

`conflicting_evidence.source_conflicts` 必须包含所有多源冲突，供 Evidence/Radar Detail 展示。

## P5 UI 要求

这张任务卡只要求后端输出结构，但要为 UI 留好字段。

Evidence / Radar Detail 后续展示：

- 当前 selected source。
- 其他 fallback / cross_check source。
- 冲突严重度。
- 数值差异。
- 为什么选当前源。

## DoD

- `historical_window(metric_id)` 改为质量优先仲裁。
- 同 fresh 状态下，质量更高的源优先。
- 主源 stale/expired 时，fresh fallback 自动接管。
- `candidates.role` 能区分 selected/fallback/cross_check。
- Radar output 写入 `conflicting_evidence.source_conflicts`。
- P1-C22 复跑仍能列出多源冲突，但 selected source 符合质量优先规则。
- 测试覆盖：
  - 同 fresh 时质量更高者胜出。
  - 质量相同时 source priority 胜出。
  - selected stale 时 fresh fallback 接管。
  - conflict 被写入 Radar output。
