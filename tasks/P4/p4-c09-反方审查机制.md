## 2026-05-21 全链条对齐补充

本卡必须对齐 P4-C12。反方审查是最终发布前硬门禁：

- 检查是否忽略 P3 triggered / near-trigger invalidation。
- 检查是否绕过 `run_mode_integrity_invalidation`。
- 检查是否存在无 evidence_id 的判断。
- 检查是否输出交易建议。
- 审查失败必须写入 `adversarial_reviews`，并设置 `publish_allowed=false` 或要求修正。

## 2026-05-21 Agent 化重构补充

反方审查必须覆盖完整 Agent workflow，而不只审查最终摘要：

- 检查 4 个 Analyst Agent 是否越权消费了不属于自己的 evidence slice。
- 检查 P4-C07 交叉质询中提出的高严重度 challenge 是否被 Judge Agent 正确处理。
- 检查 Judge Agent 是否选择性忽略少数派强反证。
- 检查 runtime guardrails 是否有失败、重试或降级。
- 检查最终总控 JSON 是否保留 `blocked_by / confidence_discount / publish_allowed`。

反方审查可以要求回退到 P4-C07 重新质询，或直接阻断发布。

## 2026-05-21 执行记录

已实现 P4 反方审查门禁，并接入当前 Agent 化链条：

- 新增 `onlybtc.p4.adversarial_review.run_adversarial_review`。
- 新增 CLI：`p4-adversarial-review --debate-id <debate_id>`。
- 审查读取最新 `judge_syntheses`、`llm_challenges`、`llm_model_votes`。
- 审查规则覆盖：
  - Judge 和 Analyst 输出必须保留具体 `evidence_ids`；
  - `blocked_by`、`run_mode_integrity_invalidation`、`confidence_discount` 不可被绕过；
  - material cross-exam challenge 必须进入 `minority_objections` 或 evidence 汇总；
  - analyst vote 不允许无 evidence；
  - 自由文本不允许交易建议类术语；
  - 硬约束存在时禁止过高置信度。
- 审查结果写入 `adversarial_reviews`。
- 审查失败时回写 `llm_debates.publish_allowed=false`。
