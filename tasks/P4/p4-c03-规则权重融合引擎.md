## 2026-05-21 全链条对齐补充

本卡必须对齐 P4-C12。规则权重融合必须先于 LLM 执行：

- 输入来自 P2 `radar_outputs` / `module_json_outputs` 与 P3 `algorithm_alerts` / `invalidation_events`。
- 融合输出作为 pre-LLM baseline，包含 module weights、signal contribution、confidence discount、risk constraints。
- 反证、data quality、run_mode integrity 必须先在规则层约束，不允许 LLM 覆盖。
- suppressed conflict 只解释不强扣分；true conflict 必须进入 confidence discount。
- 输出必须可写入后续 Evidence Pack 与 Judge Synthesis payload。

## 2026-05-21 P4-C13 全量 Radar 融合补充

规则融合层必须消费完整 Radar features，而不是只消费模块顶层信号：

- `primary_signal` feature 参与方向和强度贡献。
- `supporting_context` feature 只调整 confidence 或解释，不单独制造方向。
- `risk_context / event_context` feature 进入 risk constraints、publish impact 和状态机门禁。
- `quality_context / audit_context` feature 进入 confidence discount、missing evidence 和审计说明。
- P3-C14 的 `event_summary / daily_watch / publish_impact` 必须先在规则层形成 event constraints，再交给 LLM 解释。
- 若 `radar_modules_consumed_count < 14` 或 `uncovered_metric_count > 0`，P4 baseline 必须降低 confidence，并在审计报告中暴露。

## 2026-05-21 Agent 化重构补充

规则融合层是 Analyst Agent 与 Judge Agent 的前置约束，不是事后解释。P4-C03 输出的 pre-LLM baseline 必须进入：

- 4 个 analyst input 的 `global_context`。
- P4-C07 交叉质询的 challenge 候选。
- P4-C08 Judge Agent 的 hard/soft constraints。
- P4-C16 审计 HTML 的 baseline 区块。

LLM/Agent 可以解释规则基线，但不能删除或覆盖 hard constraints。
