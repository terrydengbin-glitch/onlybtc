# P3-C11 P1/P2/P3 全链条全量重跑与 P3 审计报告

## 状态

DONE

## 所属 Phase

P3 算法敏感检测与预警系统 / P2 Radar 质检 / P1-C22 数据链路审计 / P8 SQLite 落库验收

## 背景

P2-C19 已提供 P1-C22 与 P2 Radar 质检的全链条验收入口，但 P3 仍缺少独立审计 HTML。后续任何数据源、Radar 规则、P3 算法、预警等级、反证条件、事件窗口或 SQLite 契约变更，都必须能一键复跑并产出三份本轮独立报告：

- P1-C22 数据链路审计 HTML
- P2 Radar 质检 HTML
- P3 算法敏感检测与预警审计 HTML

## 目标

新增全链条全量重跑测试任务，统一执行：

```text
P1 live collect + P1-C22 audit
  -> P2 full radar quality audit
  -> P3 full algorithm pipeline
  -> P3 independent audit HTML
```

要求三份 HTML 都必须对应本次 run，而不是复用旧报告或旧 run。

## 交付要求

### 1. 全链条命令入口

新增 CLI / 脚本入口，例如：

```powershell
.\.venv\Scripts\python -m onlybtc.cli p3-full-audit
powershell -ExecutionPolicy Bypass -File scripts\p3-full-audit.ps1
```

支持复用最新 live 数据：

```powershell
.\.venv\Scripts\python -m onlybtc.cli p3-full-audit --no-collect-live
powershell -ExecutionPolicy Bypass -File scripts\p3-full-audit.ps1 -NoCollectLive
```

### 2. P1 / P2 / P3 三份 HTML

必须生成：

- `reports/p1-c22-真实数据全链路验收报告.html`
- `reports/p2-radar-quality-report.html`
- `reports/p3-algorithm-audit-report.html`

### 3. P3 独立审计 HTML 内容

报告必须覆盖：

- P3 run id
- run mode / non_production
- P1-C22 HTML 路径
- P2 Radar 质检 HTML 路径
- P3 pipeline summary：
  - feature metric count
  - feature rows written
  - anomaly count
  - divergence count
  - module invalidation count
  - global invalidation count
  - event window count
  - alert candidates / created / updated
- SQLite contract：
  - `p3_feature_engine` feature rows
  - `p3_anomaly_engine` rows
  - `p3_divergence_engine` rows
  - `p3_event_window_engine` rows
  - `invalidation_events`
  - `algorithm_alerts`
  - `alert_events`
- anomaly / divergence / invalidation / alert 明细
- run_mode 污染防护状态
- not_enough_samples 统计与说明

### 4. 信息同步要求

以后任何新增、优化、修复，都必须同步：

- 开发需求 / 开发文档
- 所属 Phase
- 对应任务卡
- `task index.md`

如果是新增能力，新增任务卡；如果是已有任务卡范围内修复，追加执行记录。
