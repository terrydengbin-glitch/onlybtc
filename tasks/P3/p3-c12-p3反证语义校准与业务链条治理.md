## 已发现问题

### 1. Business Recency 未进入 P3 模块反证

P2 Radar 已输出：

- `business_lagging_metrics`
- `stale_metrics`
- `expired_metrics`
- `low_coverage`

当前 P3 `check_module_invalidations()` 只对以下情况触发或近触发：

- `data_quality == low`
- `expired_metrics` 非空
- `data_quality == medium`
- `low_coverage == true`
- `source_conflicts` 非空

这意味着 `business_lagging_metrics` 即使存在，也只通过 Radar quality 间接影响，不会在 P3 invalidation 明细中形成独立反证。当前 run 中 `trade_structure_flow` 出现 `taker_buy_sell_ratio` business lagging，但模块仍为 high quality，反证为 `not_triggered`。

### 2. bullish / bearish 总控反证未区分方向

当前 `check_global_invalidations()` 中：

- `bullish_state_invalidation`
- `bearish_state_invalidation`

都使用相同的模块触发数量规则：

```text
len(triggered_modules) >= 2 -> triggered
triggered_modules or len(near_modules) >= 2 -> near_trigger
```

这会导致后续一旦多个模块级反证触发，bullish 与 bearish 两个方向可能同时触发或同时 near-trigger，语义上不够精确。

### 3. run_mode mixed history 只限制 critical，不进入反证事件

P3-C10 要求发现混合历史时阻断 critical。当前逻辑已在 alert payload 中写入：

```yaml
critical_blocked_by:
  - run_mode_mixed_history
```

但该风险没有进入 `invalidation_events`，P5 Invalidation 页或 P4 总控若只消费反证表，会看不到这条生产阻断原因。

### 4. 反证明细缺少统一严重度与触发原因字段

当前 module/global invalidation payload 有 evidence，但不同条件的 evidence 结构不一致。后续 P5 / P4 消费时，需要稳定字段：

```yaml
severity
reason_code
affected_metrics
direction_scope
source_run_id
quality_impact
publish_impact
```

## 治理目标

让 P3 反证从“通用质量门禁”升级为“业务语义明确的削弱/推翻条件”：

```text
P1 metric/source status
  -> P2 radar quality + invalidation_signals
  -> P3 module invalidation
  -> P3 global invalidation
  -> P3 alert level / critical block
  -> P4/P5 可解释消费
```

## 修复要求

### 1. 模块级反证扩展

`check_module_invalidations()` 应识别：

- `business_lagging_metrics`
- `stale_metrics`
- `expired_metrics`
- `missing_metrics`
- `source_conflicts`
- `provider_required_metrics`

建议语义：

- expired 核心指标：`triggered`
- stale 核心指标：`near_trigger`
- business lagging 且非 expected_lag：`near_trigger`
- provider_required 缺失：默认不触发，但写入 `watch_note`
- true source conflict：`triggered`
- suppressed / fallback / definition conflict：默认不触发，但保留 explain evidence

### 2. 总控反证按方向拆分

`bullish_state_invalidation` 与 `bearish_state_invalidation` 应基于 Radar signal / anomaly direction / divergence direction 判断：

- bullish 反证：宏观压力、杠杆过热、资金流转弱、价格上行但风险指标恶化等。
- bearish 反证：风险资产转强、流动性改善、资金流回流、链上/采用率改善等。

不能仅用模块反证数量同时驱动两个方向。

### 3. run_mode 风险进入反证事件

新增或扩展总控反证：

```text
run_mode_integrity_invalidation
```

当 `run_mode_risk.production_blocker == true` 时：

- 写入 `invalidation_events`
- action: `block_critical_publish`
- alert 仍可降级为 warning
- payload 包含 mixed counts 与 affected metric count

### 4. P3 审计报告同步扩展

`reports/p3-algorithm-audit-report.html` 应展示：

- run_mode integrity invalidation
- business recency invalidation 明细
- direction-specific global invalidation 结果
- true conflicts 与 suppressed conflicts 分离

### 5. 回归测试

必须覆盖：

- business lagging 核心指标触发 near-trigger。
- provider_required 缺失不触发强反证。
- true source conflict 触发模块反证。
- suppressed conflict 不触发强反证。
- bullish 与 bearish 总控反证不会因同一批模块质量问题同时触发。
- run_mode mixed history 写入 invalidation event，并阻断 critical。

## DoD

- bullish / bearish 总控反证具有方向语义。
