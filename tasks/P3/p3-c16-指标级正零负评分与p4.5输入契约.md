# P3-C16 指标级正/零/负评分与 P4.5 输入契约

## 状态

DONE

## 所属 Phase

P3 状态机、风险与事件窗口 / P4.5 Radar Scored Analyst Writer 前置契约

## 背景

P4 旧链路把大量判断留给 LLM/Agent/Judge，导致文章输出容易变成审计摘要，难以稳定产出专业趋势洞察。新的 P4.5 方案要求 P3 先把每个指标量化为正分、零分、负分，并生成一句话解释，P4.5 只负责基于这些评分与解释写专业中文评论。

## 任务目标

为每个进入 Radar/P4.5 的指标生成稳定的 `ScoredMetricEvidence`，同时为每个 Radar 板块生成 `ScoredRadarModule`：

- 当前值。
- 所属 Radar 模块。
- 方向：bullish / bearish / neutral / mixed / unavailable。
- 数值分：正、零、负。
- 分桶：positive / zero / negative / unavailable。
- 一句话指标解释。
- 本轮评分原因。
- 历史上下文。
- 数据质量与 run scope。

P4.5 消费时必须先看到 Radar 板块总分和方向，再展开该板块下的指标明细。这样 LLM 不需要自己从零推断模块方向，而是在确定性评分基础上解释“为什么这个板块 bullish / bearish / mixed”。

## 业务原则

- P3 负责量化，P4.5 负责解释。
- 每个指标的分数必须来自确定性规则、历史窗口、Radar 权重或已有 P2 方向定义。
- 每个 Radar 板块必须有总分、方向、强度、置信度和正/负/零/不可用指标分布。
- `zero` 不等于无用；它表示当前没有方向贡献，但可以作为观察项。
- unavailable/provider_required/fallback 必须显式进入 payload，不能混成 0 分。
- live 模式下只允许 current_run 真实数据进入生产评分，历史只作为 context。

## 输入

- P2 Radar module output。
- P2 Radar metric rules。
- P3 feature values：zscore、percentile、change、event phase、business recency。
- P1/P8 数据质量：quality_score、freshness_status、business_recency_status、run_mode。
- `collect_run_id / p2_radar_run_id / p3_run_id`。

## Radar 板块打分逻辑

Radar 板块总分由本板块下的指标级分数聚合而来：

```text
module_score = sum(metric_score * metric_effective_weight) / sum(active_weights)
```

其中：

- `positive / negative / zero` 指标进入 active weights。
- `unavailable` 指标不作为 0 分参与方向计算，但进入 `unavailable_metric_count` 和 `data_boundary`。
- fallback 或低质量指标可参与计算，但必须按 quality/freshness 折扣。
- provider_required 且无真实数据的指标默认进入 unavailable，不得拉低或拉高 module_score。

模块方向建议规则：

```text
module_score >= positive_threshold  -> bullish
module_score <= negative_threshold  -> bearish
abs(module_score) < threshold 且正负证据都弱 -> neutral
abs(module_score) < threshold 但正负证据都强 -> mixed
unavailable 占比过高 -> mixed 或 unavailable，并降低 confidence
```

模块必须输出：

- `module_score`
- `module_direction`
- `module_strength`
- `module_confidence`
- `module_quality_score`
- `score_bucket`
- 正/负/零/不可用指标数量
- top positive / top negative evidence ids
- `module_explanation`

## 实施范围

1. 定义 `ScoredMetricEvidence` schema。
2. 定义 `ScoredRadarModule` schema。
3. 基于 P2 Radar rule 和 P3 feature 生成 `metric_score`。
4. 聚合指标级分数，生成 Radar 板块 `module_score / module_direction / module_confidence`。
5. 定义指标与模块共用的 `score_bucket`：
   - `positive`: 分数 > 阈值。
   - `negative`: 分数 < -阈值。
   - `zero`: 数据可用但当前方向贡献接近 0。
   - `unavailable`: 数据不可用、provider_required 或 current_run 缺失。
6. 为每个指标生成 `metric_explanation`。
7. 为每个指标生成 `score_reason`。
8. 为每个 Radar 板块生成 `module_explanation`。
9. 将输出写入 P3 JSON/SQLite/HTML，供 P4.5 消费。
10. P3 HTML 增加“Radar 板块评分表”和“指标评分表”。

## 验收标准

- P3 输出覆盖 P2 Radar 涉及的全部指标。
- 每个可用指标都有 `metric_score` 和 `score_bucket`。
- 每个 Radar 板块都有 `module_score / module_direction / module_confidence`。
- 每个 Radar 板块能展示正分、负分、零分、不可用指标数量。
- 每个指标都有一句中文 `metric_explanation`。
- 每个 Radar 板块都有一句中文 `module_explanation`。
- unavailable/fallback/provider_required 不被误记为 zero。
- P4.5 可以不读取旧 P4 evidence pack，直接消费 P3 scored evidence。
- P3 审计 HTML 能展示 Radar 板块总分，以及正分、负分、零分、不可用指标列表。
