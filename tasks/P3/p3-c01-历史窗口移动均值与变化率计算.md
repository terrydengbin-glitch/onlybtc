## 计算逻辑

P3-C01 不直接抓 provider，不重新做数据源选择，只读取 P1/P8 的统一数据：

1. 调用 `historical_window(metric_id)` 获得 P1 多源仲裁后的 selected source。
2. 从 `metric_values` 读取该 `metric_id/source_id` 的历史窗口。
3. 计算基础特征。
4. 写入 `feature_values`，`module_id = p3_feature_engine`。

## 输出 Feature

每个指标默认写入：

```yaml
features:
  - latest_value
  - change_latest
  - change_window
  - ma_7
  - ma_30
  - volatility
  - slope_per_hour
  - freshness_weighted_change
  - sample_count
```

每条 feature 的 `metadata_json` 包含：

```yaml
traceability:
  metric_id: btc_price
  source_id: binance-btcusdt
  source_run_id: collect-...
  latest_ts: ISO8601
  sample_count: number
  quality_score: number
  effective_quality_score: number
  source_freshness: fresh/stale/expired
  collection_freshness_status: fresh/stale/expired
  business_recency: current/lagging/outdated/unknown
  freshness_policy: object
  selected_reason: string
  candidates: list
  conflict: object
  p3_quality_rule: normal/watch_or_info_only/evidence_only_no_sensitive_trigger/reduced_short_term_sensitivity
```

## 数据质量规则

- `expired` 数据只进入 evidence，不参与敏感触发。
- `stale` 数据只能支持 `watch/info`，不允许直接触发 `critical`。
- `business outdated` 的日频数据不算采集失败，但降低短期敏感度。
- 多源冲突沿用 P1 selected source，同时把 `conflict` 透传到 feature metadata，供 Evidence / Radar Detail 展示。
