# onlyBTC 业务链路对齐矩阵

## 1. 技术主链路

```text
Vue3 页面
  -> FastAPI 页面聚合 API
  -> Service / Repository
  -> SQLite
  -> Pipeline 写入
  -> History / Replay 复盘
```

核心约束：

- Vue3 不直接读 SQLite。
- FastAPI 不直接暴露数据库表，必须输出页面 DTO。
- Pipeline 所有关键产物先写 SQLite，再由 API 聚合读取。
- History / Replay 只读 snapshot，不读当前实时状态。
- 所有可点击对象必须带可追溯 id。

## 2. Phase 顺序

```text
P0 工程底座
  -> P8 SQLite 数据库与持久化层
  -> P1 数据源与历史数据
  -> P2 全量雷达模块
  -> P3 算法敏感检测与预警
  -> P4 LLM 推理与总控融合
  -> P9 FastAPI 页面聚合 API
  -> P5 Vue3 Dashboard 全量可视化
  -> P6 文章、历史记录与回测评分
  -> P7 动态校准与生产化
```

## 5. 已收口的重叠点

| 原任务 | 风险 | 处理结果 |
|---|---|---|
| P0-C07 数据源注册表与任务状态表设计 | 可能与 P8 重复建表 | 改为定义字段语义和枚举，表结构由 P8 实现 |
| P1-C08 历史数据存储与窗口计算基础字段 | 可能与 P8-C04 重复设计时序表 | 改为消费 P8 metric_values，负责窗口语义和特征字段 |
| P0-C09 Run Once 空流程 | 可能临时状态无法迁移 | 明确 P0 可内存模拟，P8-C05 后切换 SQLite |

## 6. 开发验收总规则

- 任一 Vue3 页面开发前，必须确认对应 P9 API 和 P8 表域。
- 任一 P9 API 开发前，必须确认 P8 Repository 可查询。
- 任一 Pipeline 阶段完成后，必须能在 Run Logs 看到 run_id 和 stage。
- 任一判断结果必须能追溯到 Evidence、数据质量和 snapshot。
- 任一历史回放必须冻结当时数据，不使用当前实时状态。
- 任一人工动作必须写入 audit_logs 或对应事件表。

## 7. Path Resolver 对齐

Resolver 不单独拆成大 Phase，统一收口到 P0-C10。所有后续 Phase 只消费 resolver，不重复设计路径规则。

| 使用场景 | Resolver 路径 | 相关任务 | 验收重点 |
|---|---|---|---|
| SQLite 数据库 | `sqlite_db_path`、`backup_dir` | P0-C10、P8-C01、P8-C11、P8-C13 | 不硬编码数据库路径，支持迁移和备份恢复 |
| 日志与 Run Logs | `logs_dir` | P0-C10、P8-C05、P9-C07 | CLI、Worker、FastAPI 日志位置一致 |
| Playwright 抓取 | `screenshots_dir`、`playwright_artifacts_dir` | P0-C10、P1-C06、P7-C05 | 截图、trace、html dump 可追溯 |
| 文章与导出 | `exports_dir` | P0-C10、P6-C02、P9-C03 | PDF、文章导出、历史报告路径可配置 |
| 前端静态资源 | `static_assets_dir`、`ui_references_dir` | P0-C10、P5-C20、P9-C01 | 打包后资源定位不依赖开发目录 |
| Mock / Seed data | `seed_data_dir`、测试临时 data root | P0-C10、P8-C12、P8-C13、P9-C13、P5-C21 | 测试数据和生产数据隔离 |

## 8. Phase DoD 门禁对齐

每个大 Phase 必须通过本 Phase 的 Mock / 测试 / DoD 任务后，才能进入下一 Phase。

| Phase | DoD 任务 | 通过后允许进入 | 必须证明 |
|---|---|---|---|
| P0 | P0-C11 | P8 | 工程骨架、CLI、FastAPI、Vue3、Worker、Resolver、Run Once mock 可运行 |
| P8 | P8-C13 | P1 | SQLite schema、migration、seed、repository、备份、路径 resolver 可用 |
| P1 | P1-C10 | P2 | 数据源 mock、清洗、fallback、source health、历史窗口可用 |
| P2 | P2-C15 | P3 | 全量雷达在典型状态下输出结构化信号、证据、冲突和降权 |
| P3 | P3-C09 | P4 | 异常检测、预警等级、冷却、反证、事件窗口可用 |
| P4 | P4-C11 | P9/P5 | Evidence Pack、多模型讨论、主裁判、反方审查、最终 JSON 可校验 |
| P9 | P9-C13 | P5 联调 | 页面聚合 API、实时推送、Run Once、错误响应、权限审计可用 |
| P5 | P5-C21 | P6/P7 | 12 个页面在 mock API 下可打开、跳转、回放、订阅实时状态 |
| P6 | P6-C07 | P7 | 文章、快照、回放、24h/72h/7D 评分闭环可用 |
| P7 | P7-C08 | 长期运行 | 校准、回滚、成本、权限、生产失败降级可用 |

## 9. 新增全局验收规则

- 所有文件路径、数据库路径、日志路径、缓存路径、导出路径、Playwright artifacts 路径必须通过 P0-C10 Path Resolver 获取。
- 所有 Mock 数据优先使用 P8 seed database 或统一 fixtures，避免每个模块各造一套样本。
- 任一大 Phase 交接前必须完成对应 Mock / 测试 / DoD 任务卡。
- DoD 未通过时，不允许把下游任务标记为可验收状态。
